# プロジェクト概要

ChampionCreator は、Pokemon Champions 向けの自動調整支援 Web アプリ。

目的は、複数の仮想敵・複数条件を同時に満たす SP 配分を探索すること。

このプロジェクトは、個人開発として長く維持できる静的 Web アプリを目指す。
短期的な便利さよりも、計算の信頼性、再現性、保守のしやすさ、公開継続のしやすさを優先する。

主に以下を扱う。

- 耐久調整: 任意の仮想敵からの攻撃を、指定回数・指定確率で耐える `H/B/D` ラインを探索する
- 火力調整: 任意の仮想敵を、指定確率で確定 1 発 / 2 発などにできる `A/C` ラインを探索する
- 素早さ調整: 任意の仮想敵や条件を抜ける `S` ラインを探索する
- 連立調整: 耐久・火力・素早さ条件を同時に満たす合法 SP 配分を探索する

このアプリは、万能なポケモン対戦支援ツールではなく、事前検証・構築検討・ダメージ確認のための計算サンドボックスとして設計する。

# 用語と前提

Pokemon Champions では、通常シリーズの `0..252 EV / 合計 508 EV` ではなく、SP（Stat Points）を正規の調整値として扱う。

ChampionCreator では次を守る。

- UI / ドメインモデル / 探索ロジックでは `SP` を正とする
- 現時点の基本前提は、各能力 `0..32 SP`、合計 `<= 66 SP`
- `EV` は `@smogon/calc` に渡すための変換表現としてだけ扱う
- `508 EV` 前提の予算判定を、ドメイン層や探索層に持ち込まない
- SP から Showdown 計算用表現への変換は、薄いアダプタに閉じ込める
- Champions 公式仕様や上流ライブラリの対応状況が変わった場合は、対応状況マトリクス、アダプタ、テストを同時に更新する

# 禁止事項
個人情報、ユーザ名、パスワード、API キーなどの機密情報をコードやドキュメントに含めないこと。

# 最重要方針

## サブエージェント
- 必要に応じて、サブエージェントを活用してください

## PROGRESS.mdについて
- プロジェクトの進捗は`PROGRESS.md`を参照し、個人用の汎用 skill progress-update を使い、PROGRESS.mdに作業内容に記録してください。 汎用 skill progress-update は ~/.codex/skills に配置されています。

## ダメージ計算エンジンを独自構築しない

このプロジェクトでは、ダメージ計算式・タイプ相性・乱数ダメージ・技威力補正などの主計算を独自実装してはいけない。

必ず Showdown 系のダメージ計算エンジンに依存する。

第一候補:

- `@smogon/calc`

既存データ層との相性が必要な場合:

- `@smogon/calc/adaptable`
- `@pkmn/data`
- `@pkmn/dex`

アプリ側で実装してよい責務は次に限定する。

- UI 入力モデル
- アプリ内ドメインモデル
- Showdown 計算エンジンへの変換
- 複数条件の評価
- SP 候補の列挙
- 探索・最適化
- 結果の整形と説明

最終的な合否判定は必ず `@smogon/calc` ベースで再評価する。

## Champions 対応の扱い

Pokemon Champions 固有の仕様・ルール・データが `@smogon/calc` や Showdown 由来データに未対応の場合、手書きでそれっぽく補完しない。

その場合は次の順で扱う。

1. 上流ライブラリで対応済みか確認する
2. `@smogon/calc/adaptable` と `@pkmn/data` 系で正式データを注入できるか確認する
3. 未対応範囲を「暫定未対応」として UI / README / テストに明記する
4. 仮対応を入れる場合も、Showdown 由来のデータ更新に差し替えやすい薄いアダプタ内に閉じ込める

計算精度の信頼性を守るため、独自の「たぶんこうだろう」計算を本筋に混ぜない。

## 対応状況マトリクス

Champions 固有仕様は、実装前・実装中・公開前に必ず対応状況を表で管理する。

分類は次を使う。

- `対応済み`: Showdown / `@smogon/calc` / `@pkmn` 系データで信頼できる計算・変換ができる
- `要確認`: 公式情報または上流ライブラリの確認が必要
- `暫定未対応`: 正確性を担保できないため、計算対象にしない
- `仮対応`: 将来差し替え可能な薄いアダプタ内に限定して扱う

初期マトリクス:

| 項目 | 初期状態 | ChampionCreator での扱い |
| --- | --- | --- |
| ダメージ計算 | 対応済み | `@smogon/calc` に委譲する |
| 技威力・タイプ相性・乱数ダメージ | 対応済み | 独自実装しない |
| 天候・フィールド・壁・ランク補正 | 要確認 | `smogonAdapter` で渡せる範囲をテストする |
| SP ルール | 対応済み | 正規モデルは `0..32 SP / 合計 66 SP` とし、変換はアダプタに閉じる |
| 使用可能ポケモン | 要確認 | 公式 / Showdown / `@pkmn` 側の対応状況を確認する |
| 使用可能技・特性・持ち物 | 要確認 | 未対応データを手書きで本筋に混ぜない |
| Mega / Tera などの Champions 仕様 | 要確認 | 対応可否を UI / README / テストに明記する |
| 連続被弾・定数ダメージ | 仮対応 | 各ヒットのダメージ生成は `@smogon/calc`、シーケンス管理だけアプリ側で行う |
| 素早さ実数値 | 要確認 | Showdown / `@pkmn` 由来データに寄せ、独自式を本筋にしない |

この表は README にも反映し、ユーザーが「何が正確に計算でき、何が未対応か」を確認できる状態にする。

# 技術方針

- TypeScript ベースで実装する
- Web アプリとしてブラウザで動く構成にする
- GitHub Pages の静的ホスティングで公開できる構成にする
- 原則としてバックエンドサーバー、データベース、実行時外部 API、実行時スクレイピングに依存しない
- アプリに必要なデータは、ビルド時にバンドルするか、静的ファイルとして同梱する
- 計算層は UI フレームワーク非依存の純粋 TypeScript に分離する
- Champions の SP モデルと Showdown 計算用表現の変換を分離する
- 重い探索は `Web Worker` で実行する
- UI コンポーネント内に探索ロジックやダメージ計算ロジックを書かない
- DOM に依存しない計算関数を作る
- 探索は中断可能にする
- 最終候補は省略なしで再検証する
- テストでは代表的な仮想敵ケースを固定し、回帰を検知できるようにする

フレームワークが未決定の場合、まずは React + Vite + TypeScript をデフォルト候補とする。ただし、既にプロジェクトで別の構成が採用されている場合は既存ルールを優先する。

# 長期保守方針

このプロジェクトでは、壊れやすい便利機能よりも、静的 Web アプリとして長く動き続けることを優先する。

## 外部依存を増やしすぎない

避けたい依存:

- 実行時外部 API
- 実行時スクレイピング
- 外部 CDN 前提の主要機能
- サーバー側処理
- ユーザーアカウント管理
- 使用率ランキング、採用率、環境データの自動取得
- HOME データなど、継続更新が必要な外部データの自動取得

このアプリは、環境分析ツールではなく、入力条件に基づく計算サンドボックスとして設計する。

## 任意数値入力を一級機能として扱う

プリセットデータは便利機能であり、アプリの本体はユーザー入力に基づく検証機能とする。

データベース未対応・新作直後・新レギュレーション直後でも使えるように、重要な値はできるだけ任意数値入力や手動上書きを可能にする。

対象例:

- HP / A / B / C / D / S の実数値
- 技威力
- 技タイプ
- 技分類
- ランク補正
- 倍率補正
- 天候、フィールド、壁、持ち物、特性などの補正条件

手入力値や上書き値を使う場合も、最終的なダメージロール生成は `@smogon/calc` ベースで行う。

## 公式画像の扱い

このプロジェクトでは、ポケモンや関連要素の公式画像・スプライトを使ってよい。

ただし、長期保守性のため次を守る。

- 画像は計算ロジックに依存させない
- 画像が読み込めなくても、計算・探索・結果表示の主要導線は壊さない
- 実行時の外部画像 URL への hotlink を主要導線にしない
- 画像の取得元、更新手順、配置場所を README またはデータ管理ドキュメントに残す
- 非公式ツールであることを README や画面上の適切な場所に明記する
- 画像差し替えや削除が必要になった場合に、UI 層だけで対応できる構造にする

## 再現性を重視する

計算条件と実行環境を後から再現できるようにする。

- アプリ画面、フッター、設定画面、または About 画面にバージョン情報を表示する
- 表示する情報は、少なくともアプリバージョン、`@smogon/calc` バージョン、データバージョンを含める
- 計算条件を JSON としてコピー / インポートできるようにする
- 可能であれば、計算条件を URL にエンコードして共有できるようにする
- 計算結果を Markdown としてコピーできるようにする

## localStorage の扱い

ユーザー設定、保存済み計算、履歴などを `localStorage` に保存する場合は、必ず `schemaVersion` を持たせる。

保存形式を変える場合は、バージョンごとにマイグレーションを用意する。

古い保存データが原因でアプリ全体が起動不能にならないように、読み込み失敗時のフォールバックも用意する。

## デプロイ方針

GitHub Pages に静的サイトとしてデプロイする。

推奨:

- GitHub Actions で自動ビルドする
- `npm ci` を使い、`package-lock.json` に基づいて依存関係を固定する
- `main` ブランチへの push でテストと静的ビルドを走らせる
- デプロイ処理は `.github/workflows/deploy.yml` に集約する

避けたい運用:

- 手元 PC から毎回手動デプロイする
- `gh-pages` ブランチを手作業で管理する
- 外部サービスや外部 API に依存したデプロイにする
- ビルド手順が README にしか存在しない状態にする

## ライセンスと第三者ライブラリ

MIT ライセンスのライブラリを使う場合でも、著作権表示とライセンス文を保持する。

`THIRD_PARTY_LICENSES.md` を用意し、少なくとも `smogon/damage-calc` と主要ライブラリのライセンスを明記する。

# ドメインモデル方針

UI 入力をそのまま計算ロジックへ渡さない。

最低限、次の概念を分離する。

- `Build`: ポケモン、レベル、性格、個体値、SP、特性、持ち物、テラスタイプなど
- `Scenario`: 仮想敵、技、天候、フィールド、壁、ランク補正、状態異常、連続行動など
- `Constraint`: そのシナリオで満たしたい条件
- `Candidate`: 探索中の SP 候補
- `Evaluation`: Showdown 計算エンジンから得た評価結果
- `Result`: UI に表示する候補と説明

耐久・火力・素早さで条件の種類は違っても、「候補を作る」「条件を評価する」「全条件を満たす候補だけ残す」という構造は揃える。

`Build` や `Candidate` が持つ配分値は、原則として Champions の `SP` とする。

`@smogon/calc` が必要とする EV 互換値、実数値、Showdown 名称などは、UI や探索層ではなくアダプタ層で変換する。

# 探索方針

## 共通ルール

- 各 SP は `0..32`
- SP は整数刻みで扱う
- 総 SP は `<= 66`
- 固定済み SP やユーザー指定の下限は必ず予算に含める
- 候補生成時点で合法な Champions SP 配分だけを列挙する
- 条件を満たさない候補はランキング前に落とす
- 合格候補だけをスコアリングする
- `508 EV` 前提の探索・枝刈り・表示を入れない
- Showdown 計算用の EV 互換値が必要な場合は、候補評価の直前にアダプタ内で変換する

## 探索保証レベル

探索モードごとに、ユーザーへ保証範囲を明示する。

- 高速モード: 粗探索 + 近傍精密探索。返す候補は必ず最終再検証済みにするが、全探索上の最適解を必ず見つけるとは限らない
- 精密モード: 合法 SP 配分の探索範囲を広げ、可能な範囲で全候補評価に近づける。全件評価した場合のみ、モデル化済み条件内で最適性を保証できる
- どのモードでも、不合格候補を結果として返さない
- UI / README では、候補の正確性保証と最適性保証を分けて説明する

## 耐久探索

`HB` と `HD` を別々に解いて合成する方式は採用しない。

必ず `H/B/D` を同時に持った 1 候補として評価する。

推奨手順:

1. 粗い代表 SP グリッドで探索する
2. 上位候補の近傍だけ合法 SP 刻みで精密探索する
3. 最終候補を `@smogon/calc` ベースで全件再検証する

耐久条件は少なくとも次を扱う。

- 何回耐えるか
- 何%以上で耐えるか
- 連続被弾
- ステルスロックなどの定数ダメージ
- 壁・天候・フィールド・急所・ランク補正

## 火力探索

`A/C` の自動調整では、攻撃側を自分、対象を仮想敵として評価する。

条件例:

- 指定技で確定 1 発
- 指定技で乱数 1 発 `x%` 以上
- 指定技で確定 2 発
- 複数対象すべてに対して必要火力を満たす

物理技は主に `A`、特殊技は主に `C` を探索対象にする。

ただし、技カテゴリや特殊仕様は独自判定せず、Showdown 側の技データ・計算結果に寄せる。

## 素早さ探索

`S` の自動調整では、指定した仮想敵・ランク・持ち物・補正条件に対して、必要な実数値または SP ラインを探索する。

条件例:

- 最速/準速の仮想敵を抜く
- ランク `+1` 後に抜く
- スカーフ込みの相手を抜く
- 追い風や麻痺など、Showdown 由来データで表現できる条件を反映する

素早さはダメージ計算と異なるが、ステータス計算や補正の信頼性を守るため、可能な限り Showdown / `@pkmn` 系データに寄せる。

## 連立探索

最終的には、耐久・火力・素早さを個別に解くだけでなく、すべての条件を同時に満たす SP 配分を返す。

連立探索では次を守る。

- `H/B/D/A/C/S` の総 SP 予算を共有する
- 固定値・下限値・探索対象を明示的に扱う
- 6 ステータスの素朴な総当たりを常用しない
- 下限ライン算出、枝刈り、予算 pruning、段階探索を使って探索空間を抑える
- 各条件の合否理由を保持する
- ボトルネック条件を結果に出す
- 条件を 1 つ満たすために別条件を壊した候補は返さない

# スコアリング方針

条件を満たした候補だけを順位付けする。

耐久候補の基本優先順位:

1. `H + B + D` が小さい
2. 総 SP の残りが多い
3. 最も厳しいシナリオに対する余裕が大きい
4. 同点なら `H` が高い候補を優先

火力候補の基本優先順位:

1. 必要な `A` または `C` が小さい
2. 他条件を壊さない
3. 乱数条件への余裕が大きい

素早さ候補の基本優先順位:

1. 必要な `S` が小さい
2. 指定条件への余裕が明確
3. 他条件を壊さない

連立候補の基本優先順位:

1. 全条件を満たす
2. 総 SP 使用量が小さい
3. 残り SP が多い
4. ユーザー指定の優先度に合う
5. 各条件の余裕が説明しやすい

# Worker 方針

探索処理は `Web Worker` に載せる。

最低限、次のメッセージを扱う。

- `start`
- `progress`
- `partialResult`
- `complete`
- `error`
- `cancel`

ユーザーが入力を変えた場合、古い探索結果が新しい UI に混ざらないように `requestId` を持たせる。

# テスト方針

最低限、次をテストする。

- `Build` が `@smogon/calc` 用オブジェクトへ正しく変換される
- SP の範囲 `0..32` と総量 `<= 66` が守られる
- SP から Showdown 計算用表現への変換がアダプタ内に閉じている
- 手入力・任意数値上書きが計算アダプタへ正しく反映される
- 天候・フィールド・壁・急所・ランク補正がラッパーに渡る
- 耐久条件の合否判定
- 火力条件の合否判定
- 素早さ条件の合否判定
- 合法 SP だけが返る
- 複数条件を同時に満たす候補だけが返る
- 最終再検証で不合格候補が落ちる
- 暫定未対応の Champions 仕様を入力した場合に、計算結果へ混ぜず明示的に扱える
- JSON インポート / エクスポートで計算条件を round-trip できる
- URL 共有で復元した条件が元の条件と一致する
- `localStorage` の `schemaVersion` とマイグレーションが動く
- バージョン情報が取得・表示できる
- 代表的な仮想敵ケースの回帰

計算結果の正しさは独自式との一致ではなく、`@smogon/calc` の結果を基準に検証する。

# 実装時の注意

- まず計算の信頼性を優先する
- 既存ファイルが増えてきたら、そのプロジェクト内の既存ルールを優先する
- 大きな設計変更をする前に、影響範囲と代替案を短く説明する
- 計算結果に関わる変更では、必ずテストか検証ログを残す
- ユーザーに確認が必要な場合は、選択肢を絞って短く聞く

# 機能追加の判断基準

採用しやすい変更:

- 静的ホストだけで完結する
- 手入力モードを強くする
- 計算条件の再現性が上がる
- 外部依存が増えない
- アダプタ層に閉じ込められる
- README やテストで説明できる
- データが古くなっても壊れにくい

慎重に扱う変更:

- 外部 API に依存する
- スクレイピングが必要になる
- 使用率データの継続更新が必要になる
- 対戦中の行動支援に近づく
- バックエンドが必要になる
- 計算エンジンの内部実装に深く依存する
- 保守しないとすぐ壊れる
